# GodsView: Persistence Boundaries

> Architecture Decision Record — Phase 118
> Every file in this repo falls into exactly one of three categories.

---

## The Three Categories

### 1. SOURCE — Lives in Git, Never Generated

These are the files humans (and Claude) write. They are the **single source of truth**.
If you delete everything else and rebuild, these are what you start from.

| Path Pattern | Examples |
|---|---|
| `api-server/src/**/*.ts` | Route handlers, lib modules, types |
| `artifacts/godsview-dashboard/src/**/*.tsx` | React pages, components, hooks |
| `lib/*/src/**/*.ts` | Shared packages (db, api-zod, strategy-core) |
| `services/**/*.py` | Python v2 microservices |
| `godsview-openbb/**/*.py` | OpenBB integration layer |
| `docs/**/*.md` | Architecture docs, runbooks, specs |
| `scripts/**/*.sh` | Deployment, cleanup, verification |
| `*.json` (root configs) | package.json, tsconfig, docker-compose |
| `Dockerfile`, `nginx/**` | Infrastructure definitions |
| `.github/workflows/*.yml` | CI/CD pipeline definitions |
| `.env.example` | Environment template (no secrets) |

**Rule:** Everything in this category is committed. PRs touch these files.

---

### 2. STATE — Lives on Disk at Runtime, Never Committed

These are produced by the running system. They are **ephemeral** and must be
reconstructible from source + environment.

| What | Where | Lifecycle |
|---|---|---|
| PostgreSQL data | Docker volume `godsview_postgres_data` | Persists across restarts, backed up separately |
| Guard state JSON | `*_guard_state*.json` | Written by safety engines, gitignored |
| Runtime logs | stdout / Winston JSON | Shipped to monitoring, rotated |
| Brain sync artifacts | `_brain_sync.*` | Temporary transfer format, gitignored |
| Session/trade state | In-memory + DB | DB is authoritative, memory is cache |
| ML model artifacts | `services/ml/models/` | Trained at runtime, versioned in MLflow |
| LanceDB vector store | `services/memory/data/` | Rebuilt from embedding pipeline |
| Prometheus metrics | `monitoring/data/` | Time-series, separate retention policy |

**Rule:** None of this goes in git. If it appears in `git status`, something is misconfigured.
`.gitignore` is the enforcement layer.

---

### 3. BUILD — Generated from Source, Disposable

These are deterministic outputs of the build process. Given the same source and
dependencies, you get the same build artifacts.

| What | Generated By | Path |
|---|---|---|
| API server bundle | `esbuild` via `pnpm build` | `artifacts/api-server/dist/` |
| Dashboard static assets | `vite build` | `artifacts/godsview-dashboard/dist/` |
| TypeScript declarations | `tsc --declaration` | `*.d.ts` alongside source |
| `node_modules/` | `pnpm install` | Workspace root + package dirs |
| Python `__pycache__/` | Python interpreter | Beside `.py` files |
| Docker images | `docker build` | Container registry (GHCR) |
| `.tsbuildinfo` | TypeScript incremental | Workspace root |

**Rule:** These should be gitignored (and they are). The CI pipeline rebuilds them from source.
Exception: `artifacts/` is committed because the dashboard build output is served directly by nginx.

---

## The Decision Tree

When you create or encounter a file, ask:

```
Is it written by a human/AI?
  YES → SOURCE (commit it)
  NO  → Is it produced by the build toolchain?
          YES → BUILD (gitignore it, CI rebuilds)
          NO  → STATE (gitignore it, infra manages it)
```

---

## Why `artifacts/` Is Committed

The `artifacts/` directory is an intentional exception to the BUILD rule.
GodsView uses a **pre-built deployment model** where:

1. The dashboard is built locally or in CI
2. The built output is committed to `artifacts/godsview-dashboard/dist/`
3. Nginx serves it as static files from the Docker image
4. The API bundle lives in `artifacts/api-server/dist/`

This means `git clone` → `docker compose up` works without a build step on the
deployment target. The tradeoff is larger commits when the dashboard changes,
but it eliminates build-time dependencies in production.

**If we move to a registry-based deployment** (push Docker images to GHCR, pull
on deploy), artifacts should be removed from git and added to `.gitignore`.

## Boundary Violations to Watch For

| Violation | Symptom | Fix |
|---|---|---|
| State leaking into git | `*_guard_state*.json` in `git status` | Add to `.gitignore`, `git rm --cached` |
| Secrets in source | `.env` tracked, API keys in code | `git rm --cached .env`, rotate keys |
| Build artifacts growing | `artifacts/` diff is 50MB+ | Consider registry-based deploy |
| Source generated by tools | Auto-generated code not marked | Add `// @generated` header, exclude from lint |
| node_modules committed | Repo clone is 500MB+ | Verify `.gitignore` has `node_modules` |
| Runtime logs in repo | `*.log` files appearing | Add `*.log` to `.gitignore` |

## Enforcement

| Layer | Mechanism |
|---|---|
| `.gitignore` | Prevents accidental staging of state + build files |
| `scripts/clean-repo.sh` | Automated scan and cleanup with `--dry-run` |
| `scripts/verify-phase.sh` | 10-check gate including secret scan and file inventory |
| CI pipeline (Gate 1) | TypeScript typecheck blocks merge on type errors |
| CI pipeline (Gate 2) | Test suite blocks merge on failures |
| Docker multi-stage | Only production deps + built artifacts enter the final image |
| `CODEOWNERS` (future) | Require review for changes to infra files |

## Quick Reference

```
git (source of truth)
├── api-server/src/       ← SOURCE
├── artifacts/dist/       ← BUILD (committed exception)
├── lib/*/src/            ← SOURCE
├── services/             ← SOURCE
├── docs/                 ← SOURCE
├── scripts/              ← SOURCE
├── .github/              ← SOURCE
├── Dockerfile            ← SOURCE
└── .env.example          ← SOURCE (no secrets)

disk only (never committed)
├── .env                  ← STATE (secrets)
├── *_guard_state*.json   ← STATE (runtime)
├── node_modules/         ← BUILD (pnpm install)
├── __pycache__/          ← BUILD (python)
├── *.tsbuildinfo         ← BUILD (tsc cache)
├── postgres data volume  ← STATE (database)
└── monitoring/data/      ← STATE (metrics)
```
