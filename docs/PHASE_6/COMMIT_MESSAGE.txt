phase 6: production hardening — reliability, observability, security

Strictly operational hardening. No new product surface, no strategy
changes, no execution-pipeline changes, no risk-pipeline changes, no
proof or reconciliation changes. Adds the safety properties needed to
run continuously in paper mode without requiring a human to babysit
the process.

================================================================
RELIABILITY
================================================================

* Fail-fast environment validator (lib/ops/phase6_env.ts).
  - Asserts DATABASE_URL, GODSVIEW_OPERATOR_TOKEN, ALPACA_API_KEY,
    ALPACA_SECRET_KEY, REDIS_URL are non-empty before app.listen().
  - Demo mode permits broker keys + REDIS_URL to be empty; everything
    else is required in every mode.
  - On missing variables: pino.fatal one line per missing var,
    process.exit(1). Combined with `restart: always`, a misconfigured
    container surfaces immediately as a visible restart loop.

* Pure timeout + retry helper (lib/ops/with_retry.ts).
  - withRetry(fn, { timeoutMs, maxRetries=2, backoffMs, shouldRetry }).
  - Exponential backoff between attempts.
  - Per-attempt abort via Promise.race timer.
  - Reports actual attempts on failure (not maxRetries+1) so a
    shouldRetry short-circuit produces an honest count.
  - Available infrastructure. Used by /api/health/phase6 for its DB
    check. Not yet wired into the existing placeOrder path — that
    would be a refactor of the execution layer, which Phase 6
    explicitly forbids; deferred.

* docker-compose.minimal.yml restart policy: unless-stopped → always.
  Container will be restarted by the Docker daemon on any exit,
  including exits caused by the Phase 6 fail-fast env validator.

================================================================
OBSERVABILITY
================================================================

* In-process counters (lib/ops/counters.ts) — pure module:
    total_requests, failed_requests,
    order_attempts, order_executions, reconciliation_runs.

* Counter observer middleware (lib/ops/counter_middleware.ts).
  - Mounted as the FIRST middleware in routes/index.ts.
  - Observes responses; never modifies request or response.
  - Increments order_attempts on POST /api/alpaca/orders entry,
    order_executions on 2xx exit, failed_requests on >=400.
  - reconciliation_runs increments via a separate one-line wrapper
    on /api/proof/reconciliation/run.

* Phase 6 read endpoints (routes/phase6.ts):
  - GET  /api/health/phase6     service / db / redis / last reconciler /
                                 last data-health timestamps
  - GET  /api/ready/phase6      strict readiness (db + redis + env)
  - GET  /api/ops/metrics       JSON counters; rejected_trades sourced
                                 from persisted execution_audit so the
                                 count survives process restarts.
  - 200/503 status codes reflect the underlying state.
  - These do NOT replace the existing /healthz, /readyz, /metrics
    endpoints; they sit alongside.

* Structured logging contract documented and verified:
  every log line includes time, level, channel, request_id, and
  audit_id when trade-related (Phase 5 child loggers + pinoHttp
  request id + Phase 3 audit log fields).

================================================================
SECURITY
================================================================

* Operator-token gate added to POST /api/proof/reconciliation/run
  via routes/index.ts (requireOperator middleware mounted at the
  route prefix).
  - 403 operator_token_required if token not configured.
  - 401 unauthorized if token absent on request.
  - 403 forbidden if token present but wrong (timing-safe compare).
  - reconciliation_runs counter increments only AFTER auth passes.

* Per-route rate limits stacked on top of the existing app-wide
  /api limiter (300/min):
    POST /api/alpaca/orders            30/min  (default)
    /api/proof/*                      120/min  (default)
    POST /api/proof/reconciliation/run  6/min  (default)
  All three are env-overridable
  (GODSVIEW_PHASE6_ORDERS_RATE_LIMIT_PER_MIN,
   GODSVIEW_PHASE6_PROOF_RATE_LIMIT_PER_MIN,
   GODSVIEW_PHASE6_RECONCILE_RATE_LIMIT_PER_MIN).
  Limits are per-IP, per-method; standard X-RateLimit-* headers
  returned. Prevents accidental retry loops, abuse, and
  reconciler-trigger flooding.

================================================================
HARD GUARANTEES
================================================================

* No code in any of the protected subsystems is touched:
  - lib/strategies/ob-retest-long-1h/             (Phase 2)
  - lib/order_executor.ts                          (Phase 3)
  - lib/risk/risk_pipeline.ts                      (Phase 3)
  - lib/paper_trades/{store,reconciler,integrity,jobs}.ts  (Phase 4-5)
  - routes/paper_proof.ts                          (Phase 4-5 endpoints unchanged)
  - The existing /healthz (app.ts) and /readyz, /metrics
    (routes/health.ts) endpoints                  (Phase 1 baseline)

* The Phase 6 endpoints live at distinct paths
  (/api/health/phase6, /api/ready/phase6, /api/ops/metrics) so they
  can be inspected, deployed, and aliased independently of the
  existing endpoints.

* Phase 6 fail-fast env validator runs after the existing
  validateEnv() call. If the existing validator is misconfigured for
  some other reason it still fires first; this validator only adds
  the explicit Phase 6 contract on top.
