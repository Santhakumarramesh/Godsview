Here’s a practical upgrade roadmap for **Godsview** that takes it from “smart prototype” to “serious trading system.”

## Phase 0 — Stabilize the base

Goal: make the repo trustworthy before adding more features.

What to do:

* Add a proper `README.md` with architecture, setup, commands, supported instruments, and current status.
* Fix the **tests so they match the current pipeline API**. Right now the code and tests look out of sync, which is your biggest immediate weakness. The live pipeline uses `TradingDecision.decision_type`, `bar_ts`, `scores`, and `submit_bracket()`, while the test file expects older fields and flows.  
* Add a `.env.example` for Databento and Anthropic keys.
* Add GitHub repo metadata: description, topics, maybe a banner image.
* Add a `requirements.txt` or make the packaging/install path very obvious if `pyproject.toml` is already used.

Deliverables:

* Clean repo landing page
* Passing tests
* One-command local setup
* Clear “paper trading only” label

Exit criteria:

* A new person can clone, install, and run `replay` mode without confusion.
* Tests pass in CI.

---

## Phase 1 — Visibility and diagnostics

Goal: stop hidden failures and make the system observable.

What to build:

* Add `python run.py doctor`
* Print:

  * config loaded
  * model directory found or missing
  * Claude active / stub / disabled
  * recall engine active
  * detector active
  * broker active
  * data feed credentials available
* Add structured startup logs and a final “system readiness” summary.
* Add a health status object per session showing whether each layer is live, degraded, or bypassed.

Why this matters:
Your current architecture degrades gracefully, which is good, but it can silently fall back to neutral or stub behavior. That is dangerous unless surfaced very clearly.  

Deliverables:

* `doctor` command
* readiness report JSON
* session startup summary in logs

Exit criteria:

* You always know exactly which parts are real and which parts are fallback.

---

## Phase 2 — Fix the research loop

Goal: make model training and evaluation believable.

What to build:

* Standard training manifest for every run:

  * setup type
  * sample count
  * features used
  * validation metrics
  * calibration metrics
  * saved model path
* Add walk-forward evaluation, not just random CV.
* Save:

  * ROC
  * precision-recall
  * calibration curve
  * confusion matrix
  * threshold analysis
* Add per-setup reports for:

  * absorption reversal
  * sweep reclaim
  * continuation
  * cvd divergence
  * breakout failure 

What is missing now:
The pipeline design is strong, but you need much stronger proof that each setup has edge and under what conditions. Right now the repo feels more engineered than validated.

Deliverables:

* `training_runs/<timestamp>/manifest.json`
* performance charts
* setup ranking report
* feature importance report

Exit criteria:

* You can answer: which setup works best, in which regime, in which session, and with what threshold.

## Phase 3 — Improve execution realism

Goal: make paper trading more realistic.

What to upgrade in `paper_broker` and execution:

* Add partial fills
* Add latency simulation
* Add bar-gap edge case handling for entries and exits
* Add duplicate order protection
* Add order rejection simulation
* Add exchange disconnect / reconnect simulation
* Add reconciliation checks between intended state and broker state

Your current paper broker already has a decent base with bracket orders, slippage, and fills, which is a good start. 

Deliverables:

* execution simulation modes: simple / realistic / stress
* fill-quality report
* per-trade latency and slippage logs

Exit criteria:

* Replay and paper sessions behave closer to real execution, not idealized fills.

## Phase 4 — Add session intelligence

Goal: make the bot adaptive, not just layered.

What to add:

* Dynamic thresholds by session and regime instead of fixed values only
* Per-setup allowed sessions
* Per-setup cooldown rules
* Auto-disable weak setups when session conditions are poor
* “No-trade day” logic after repeated degradation
* Rolling confidence tracker for each setup

The risk engine already has session thresholds and regime gating, which is strong. This phase makes that adaptive instead of static. 

Deliverables:

* session policy file
* setup-session matrix
* adaptive threshold logic
* daily regime summary

Exit criteria:

* The system changes behavior intelligently across open, midday, lunch, and overnight.

## Phase 5 — Claude layer upgrade

Goal: make Claude useful without letting it become a dangerous black box.

What to do:

* Keep Claude as a filter, not a predictor.
* Log Claude’s effect separately:

  * veto rate
  * positive adjustment rate
  * average adjustment
  * win rate with Claude on vs off
* Add ablation testing:

  * ML only
  * ML + OF
  * ML + OF + Claude
* Add timeout and cost analytics
* Add stricter schema validation on Claude output
* Add prompt/version tracking

This is important because your Claude layer is one of the most interesting parts of the repo, but also the easiest place for performance illusions to sneak in. 

Deliverables:

* Claude effectiveness report
* prompt versioning
* cost per session
* veto analytics dashboard

Exit criteria:

* You can prove whether Claude helps or hurts.
