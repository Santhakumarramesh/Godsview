**TradingView** finds the area.

**ML model** scores the setup statistically.

# 1. Final combined architecture

## Layer 1 — TradingView structure layer

Use TradingView to detect:

* order blocks
* support/resistance
* VWAP position
* trend bias
* session highs/lows
* higher-timeframe bias

This answers:

**“Is price in a place where a trade is even worth considering?”**

---

## Layer 2 — Order-flow layer

Use Bookmap-style logic to detect:

* absorption
* delta shifts
* sweeps and failed sweeps
* liquidity pull/stack
* liquidity vacuum
* CVD divergence

This answers:

**“Are real buyers or sellers actually defending or attacking here?”**

---

## Layer 3 — Recall engine

This is the memory layer.

At the moment a setup appears, it should remember:

* last 20 candles on 1m
* last 20 candles on 5m
* last 20 candles on 15m
* last 20 candles on 1h
* last few seconds of order-flow behavior

It does not just store candles.
It converts them into usable features such as:

* recent returns
* candle ranges
* wick behavior
* trend slope
* distance to VWAP
* distance to swing highs/lows
* regime state

This answers:

**“What has the market been doing across multiple timeframes before this setup?”**

---

## Layer 4 — ML layer

Use XGBoost or LightGBM first.

Do not train it on all candles.
Train it only on **setup moments**.

Example target:

* label = 1 if price hits `+6 ticks` before `-4 ticks` within `45 sec`
* label = 0 otherwise

This answers:

**“Historically, when this exact setup appears with this exact context, does it usually work?”**

---

## Layer 5 — Claude reasoning layer

Claude should not replace the ML model.
Claude should sit **after** the model.

Claude takes:

* setup type
* structure summary
* order-flow summary
* recall-engine context
* ML probability

Then returns:

* adjusted probability
* quality score
* trade / no-trade
* reasoning

This answers:

**“Given the current context, should I trust this setup right now?”**

Claude is strongest for:

* regime interpretation
* anomaly detection
* context-aware rejection
* explaining failed trades
* improving your prompts and filters over time

---

## Layer 6 — Risk and execution layer

This layer blocks bad trades even if everything else looks good.

It checks:

* spread
* slippage
* news lockout
* position size
* daily loss
* duplicate signals
* stale signal timing

This answers:

**“Even if this setup is good, is execution safe right now?”**

# 2. Final decision logic

A trade should only happen when all these agree:

## Structure

Price is at a meaningful location.

## Order flow

Absorption, delta, sweep, or liquidity confirms it.

## Recall engine

Past candles across timeframes support the setup.

## ML

Probability is above threshold.

## Claude

No contextual red flags.

## Risk engine

Execution conditions are safe.

So the decision rule becomes:

```text
IF
  structure_valid
  AND orderflow_confirmed
  AND recall_context_supportive
  AND ml_prob >= threshold
  AND claude_decision == TRADE
  AND risk_engine_passes
THEN
  execute
ELSE
  skip
```

# 4. How training should work

## Step 1

Run your rules engine over historical data to find candidate setups.

## Step 2

At each candidate timestamp, ask the recall engine for multi-timeframe context.

## Step 3

Create one feature row containing:

* structure features
* order-flow features
* recall-engine features
* session/regime features

## Step 4

Label the row by target-before-stop.

## Step 5

Train separate models for each setup family.

## Step 6

Walk-forward validate only. Never random split.

That gives you a proper learning system based on your strategy, not random market prediction.
